软件需求规格说明书 (SRS)
项目名称：校园互助服务平台 (Campus Mutual Assistance Platform)
1. 引言 (Introduction)
1.1 目的 (Purpose)
本文档旨在详细描述“校园互助服务平台”的软件需求规格。本文档将作为平台设计、开发、测试以及验收的基础，面向的读者包括项目经理、UI/UX设计师、前后端开发工程师、测试工程师以及最终的平台运营方。

1.2 范围 (Scope)
“校园互助服务平台”是一个专为在校大学生设计的综合性服务社区。该平台允许学生发布和接收各类互助任务，主要涵盖：快递代取/跑腿、学习辅导、二手交易、组队匹配四大核心业务。平台旨在通过整合校园内部资源，提高学生的生活效率，促进校园内的互助互信氛围。系统将包含移动端（小程序/App）和Web端后台管理系统。

1.3 术语、缩写词和缩略语 (Definitions, Acronyms, and Abbreviations)

发布者 (Publisher): 在平台上发布需求或物品的用户。

接单者 (Receiver): 在平台上接受任务或购买物品的用户。

校园认证 (Campus Verification): 通过校园邮箱（.edu.cn）或学信网/学生证进行的实名学生身份绑定。

信用分 (Credit Score): 衡量用户履约能力和信誉的量化指标，影响接单和发布权限。

1.4 参考文献 (References)

IEEE Std 830-1998, IEEE Recommended Practice for Software Requirements Specifications.

1.5 概述 (Overview)
本文档其余部分将对平台进行总体描述（第2章），随后在第3章中详细列出所有外部接口、功能、性能和系统属性的具体需求。

2. 总体描述 (Overall Description)
2.1 产品视角 (Product Perspective)
本项目是一个全新的独立分布式系统。面向学生端采用移动优先策略（微信小程序或移动App，便于随时随地接发单），面向管理员端采用Web浏览器界面。系统依赖第三方支付接口（如微信支付/支付宝）进行悬赏资金托管，并依赖云存储服务（如OSS）存储用户上传的图片和凭证。

2.2 产品功能 (Product Functions)
平台的主要功能模块包括：

身份与安全模块： 注册登录、学生身份实名认证、信用分管理。

任务大厅模块： 需求分类展示、条件筛选、智能推荐。

订单发布与承接模块： 需求信息的填写与发布、接单与状态扭转、取消与申诉。

即时通讯模块： 买卖双方/接发单双方的站内私信沟通。

评价与反馈模块： 订单完成后的双向打分与评价。

2.3 用户特征 (User Characteristics)

在校大学生： 平台的主要使用者。通常熟悉移动互联网应用，对时间敏感，预算有限，具有强烈的社交和互助需求。

系统管理员： 平台运营人员。负责审核用户资质、处理违规内容、调解交易纠纷及平台数据分析。

2.4 约束条件 (Constraints)

合规性约束： 必须严格遵守《网络安全法》及个人信息保护法，保护学生隐私数据。

业务约束： 资金交易需符合相关金融监管规定，平台可能需要引入虚拟积分或第三方资金托管机制以避免直接沉淀资金。

技术约束： 移动端需兼容主流的 iOS 和 Android 系统版本。

2.5 假设和依赖关系 (Assumptions and Dependencies)

假设学校允许学生进行此类互助交易，且不存在校园网络封锁。

依赖于第三方短信网关进行验证码发送。

依赖于第三方地图API（可选）用于计算跑腿任务的校园内距离。

3. 具体需求 (Specific Requirements)
3.1 外部接口需求 (External Interfaces)

用户界面 (UI): 采用现代化、清新的设计风格。首页需提供四大核心业务的快捷入口，底部导航栏包含“首页、大厅、发布、消息、我的”。

硬件接口: 移动端需调用手机摄像头/相册（用于上传认证材料、商品图片、完成凭证）。

软件接口: 对接校园统一身份认证系统（若学校开放 API）或通过邮箱系统接口验证。对接微信/支付宝支付 SDK。

3.2 功能需求 (Functional Requirements)

3.2.1 用户认证与管理 (User Authentication & Management)

REQ-U-01 (注册登录): 支持手机号验证码注册，支持微信一键登录授权。

REQ-U-02 (校园认证): 用户必须通过上传学生证或验证edu邮箱完成实名认证后，方可进行发布和接单操作。

REQ-U-03 (个人中心): 用户可查看个人的发布记录、接单记录、我的收藏、信用分数及历史评价。

3.2.2 核心业务模块：发布与接单 (Core Modules: Post & Accept)

REQ-B-01 (快递跑腿):

发布者需填写：取件点、送达点、物件大小、截止时间、悬赏金额。

接单者接单后，系统隐藏取件码，仅接单者可见；完成后需上传送达图片作为凭证。

REQ-B-02 (学习辅导):

发布者可发布“求教”或“授课”需求，包含：科目名称、时间安排、时薪/一口价、授课方式（线上/线下）。

REQ-B-03 (二手交易):

卖家发布商品需上传至少1张图片，填写新旧程度、价格、交易地点（如某校区某食堂）。

买家可进行“留言询价”或“直接拍下”。

REQ-B-04 (组队匹配):

支持发布比赛组队、课题研讨、拼车出行等信息。

需填写：队伍目标、目前人数、缺编要求（如“缺一名前端开发”）。用户可提交“入队申请”，由队长审批。

3.2.3 订单生命周期管理 (Order Lifecycle)

REQ-O-01 (状态扭转): 订单状态应包括：待接单 -> 进行中 -> 待确认完成 -> 已完成 -> 已评价。

REQ-O-02 (资金托管): 发布需支付悬赏金的订单时，资金先扣除并托管在平台；接单者完成任务且发布者确认后，资金打入接单者账户。

3.2.4 站内互动 (Interaction)

REQ-I-01 (即时通讯): 支持双方发送文字、图片、语音消息，防止隐私泄露（无需添加微信即可沟通）。

REQ-I-02 (双向评价): 订单完成后，双方可在3天内对彼此进行星级（1-5星）和文字评价，评价计入历史档案。

3.2.5 后台管理系统 (Admin Dashboard)

REQ-A-01 (内容审核): 管理员可查看、下架违规的需求或商品（包含敏感词过滤）。

REQ-A-02 (纠纷仲裁): 管理员可介入处于“申诉中”的订单，查看双方聊天记录并进行资金判决。

3.3 性能需求 (Performance Requirements)

响应时间: 页面加载时间应控制在 2 秒以内；核心交互（如“抢单”按钮点击）响应时间不超过 500 毫秒。

并发处理: 系统需支持日常 1,000 QPS 的并发请求，在期末考试季（资料购买）或开学季（二手书、跑腿）等高峰期需支持 5,000 QPS。

可用性: 系统应保证 99.9% 的正常运行时间。

3.4 设计约束 (Design Constraints)

技术栈约束: 建议后端采用 Spring Boot / Node.js 架构，前端采用 Vue.js / React Native，数据库使用 MySQL 和 Redis（用于高并发抢单缓存）。

合规化: 密码必须使用强加密算法（如 bcrypt）进行哈希处理存储，不得明文保存。

3.5 软件系统属性 (Software System Attributes)

安全性 (Security): 接口必须通过 Token（如 JWT）进行身份校验；防范 SQL 注入、XSS、CSRF 等常见网络攻击；对频繁发单/接单账号有防刷单风控机制。

可维护性 (Maintainability): 代码需遵循统一的格式规范（如阿里Java开发手册/ESLint），关键业务逻辑必须有单元测试覆盖，API接口需自动生成在线文档（如 Swagger）。

可移植性 (Portability): 后端服务应容器化（Docker），确保能够快速在不同云服务器（阿里云/腾讯云）之间迁移和部署。
