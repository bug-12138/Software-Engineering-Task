# 需求规格说明书 (SRS)

## 1. 引言
### 1.1 目的
本文档旨在明确平台的功能与非功能需求。本文档的所有业务逻辑均直接来源于本团队在本校内开展的 20+ 份问卷调查及 4 份深度用户访谈，确保系统完全贴合真实的校园生态。 `[人工补充]`

### 1.2 范围
本系统旨在解决校园内“信息匹配效率低”和“陌生人互助缺乏信任”两大痛点。系统包含用户身份与信用认证、细分场景的需求发布、防纠纷的订单流转、以及双向评价体系。 `[人工补充]`

---

## 2. 总体描述
### 2.1 目标用户特征（基于访谈原始数据提取）
* **大一新生群体**：对校园环境陌生，具有“代搬行李”、“校园指引”和“买二手教材”的强需求。**痛点**：防骗意识较弱，对平台的安全性、身份真实性要求极高。 `[人工补充：源自受访者04数据]`
* **大二/大三群体（主力军）**：需求呈现高频次、碎片化特征。理工科痛点多为“实验室走不开需代取外卖/快递”；文科痛点多为“求助软件安装/重装系统”。他们是平台最核心的需求方与服务方。 `[人工补充：源自受访者01、02数据]`
* **大四考研群体**：时间敏感度极高。核心需求为“代买三餐送至图书馆”和“考研资料交易”。**痛点**：对服务响应速度和准时性要求苛刻，愿意为高效率支付更高报酬。 `[人工补充：源自受访者03数据]`

### 2.2 运行环境
考虑到学生在宿舍、实验室、图书馆、教学楼等场景的移动性，系统需要适配移动端浏览器（Web/H5）的浏览体验，同时兼容 PC 端操作。 `[人工补充]`

## 3. 功能需求

### 模块一：用户认证与信用管理 (P0)
* **REQ-U-01 (实名准入)**: 系统必须集成教务处学号或要求上传学生证，强制完成实名认证后才可使用发布与接单功能。（解决问卷反馈的“缺乏信任、怕被骗”痛点） `[人工补充]`
* **REQ-U-02 (信用分体系)**: 每个用户初始信用分为 100 分。成功履约加分，被投诉或违约扣分。信用分低于 60 分的用户将被限制接单权限。 `[人工补充]`
* **REQ-U-03 (双重身份)**: 系统不区分“商家”和“买家”账号，同一账号可随时在“需求方”和“服务方”之间无缝切换。 `[AI生成+人工修改：根据校园互助身份重叠的特性修改]`

### 模块二：场景化需求发布与大厅 (P0)
* **REQ-D-01 (细分发单表单)**: 需求发布表单除了常规的标题、描述、金额外，必须包含预设的校园专属标签（如：南区快递点、一教、图书馆等），以提高填写效率。 `[人工补充]`
* **REQ-D-02 (专属板块划分)**: 需求大厅必须明确划分以下四大板块：
  1. 跑腿代办（快递/外卖）
  2. 技能求助（装软件/修电脑）
  3. 学习互助（考研占座/资料分享）
  4. 二手闲置
  `[人工补充：严格对应4位受访者的实际需求场景]`
* **REQ-D-03 (加急悬赏机制)**: 针对考研党等时间敏感用户，发布需求时可支付额外积分/金额使用“加急”功能。加急订单在首页大厅置顶展示，并带有醒目倒计时。 `[人工补充]`

### 模块三：防纠纷的订单流转 (P0)
* **REQ-O-01 (锁定机制)**: 服务方一旦点击“接单”，该需求立即从大厅隐藏，进入“进行中”状态，防止一单多接。 `[AI生成+人工修改：强化了并发控制的业务逻辑]`
* **REQ-O-02 (双向确认闭环)**: 为防止恶意点完成，订单的结束必须经过两步：服务方点击“已送达/已完成” -> 需求方点击“确认验收”。若需求方超 24 小时未确认，系统自动验收。 `[人工补充]`
* **REQ-O-03 (异常取消机制)**: 接单后 10 分钟内允许无责取消（防止手滑）；超过 10 分钟单方面取消将扣除信用分。 `[人工补充]`

### 模块四：评价与仲裁 (P1)
* **REQ-C-01 (星级与文字双评)**: 订单完成后必须进行 1-5 星打分。 `[AI生成]`
* **REQ-C-02 (低分举证)**: 给出 3 星及以下评价时，必须强制填写不少于 15 字的原因并支持上传截图，自动生成“待仲裁”工单，防止恶意差评毁坏服务方信用。 `[人工补充]`

## 4. 非功能需求

### 4.1 性能需求 (Performance)
* **REQ-NF-P01 (响应时间)**: 在校园网环境（如人流量密集的食堂、图书馆）下，移动端需求大厅列表的刷新与数据加载时间不得超过 1.5 秒。 `[人工补充]`
* **REQ-NF-P02 (并发处理)**: 系统需支持至少 500 名用户同时在线访问及操作（主要应对午饭、晚饭等“代取外卖”高频时段），保证核心的“接单”接口不崩溃、不出现多个人同时抢到同一单的并发超卖现象。 `[人工补充]`

### 4.2 安全需求 (Security)
* **REQ-NF-S01 (隐私脱敏)**: 需求方填写的具体宿舍门牌号、联系电话等敏感隐私信息，默认必须在前端大厅进行打码隐藏；仅当服务方“成功接单后”才对其解锁可见，且订单结束后自动恢复隐藏。 `[人工补充：源自访谈中反映的隐私泄露担忧]`
* **REQ-NF-S02 (数据安全)**: 用户的登录密码必须使用强哈希算法（如 BCrypt）单向加密后存储入库；后端所有的 API 接口必须进行参数校验，防范 XSS 攻击和 SQL 注入。 `[人工补充]`

### 4.3 可用性需求 (Usability)
* **REQ-NF-U01 (移动优先适配)**: 考虑到大学生的使用场景高度依赖手机碎片化时间，前端 UI 必须优先采用移动端响应式设计，完美适配主流手机屏幕尺寸，避免出现横向滚动条或核心按键被遮挡。 `[人工补充]`
* **REQ-NF-U02 (极简交互路径)**: 核心业务流程必须符合直觉且精简。从用户打开首页到成功发布一条互助需求，页面的交互点击步骤不得超过 3 步，以降低（尤其是大一新生）的使用学习成本。 `[人工补充]`

### 4.4 可维护性需求 (Maintainability)
* **REQ-NF-M01 (架构与代码规范)**: 系统必须严格贯彻前后端分离架构（Vue3 + Spring Boot），后端代码需遵循 Controller、Service、Mapper 三层规范，并严格执行团队制定的 Git Commit 提交规范，确保后续新功能的低成本迭代。 `[人工补充]`
* **REQ-NF-M02 (接口文档自动化)**: 后端项目必须集成 Swagger/Knife4j 等自动化工具，实时生成并维护在线 API 接口文档，降低团队内部前后端联调的沟通成本。 `[人工补充]`
